# Спецификация доменных требований: шаблон школьного расписания, продленка и дополнительные занятия

## 1. Назначение документа

Этот документ фиксирует бизнес-требования и доменные инварианты, которые были сформулированы в обсуждении вокруг страницы редактирования шаблона школьного расписания.

Документ не предлагает конкретную реализацию на уровне Prisma или UI-компонентов. Его задача:

- зафиксировать, что именно должна уметь система;
- развести понятия "кому занятие положено", "кто реально его посещает", "в чьей строке оно показывается", "что считается одним физическим занятием";
- определить ограничения, которые должна учитывать будущая модель данных и интеграция с FET.

---

## 2. Контекст предметной области

Система строит и редактирует школьное расписание, в котором присутствуют:

- основная школьная программа;
- режимные активности продленки и школьного дня;
- обязательные дополнительные предметы;
- дополнительные предметы по выбору.

При этом реальная жизнь школы создает нетривиальные сценарии:

- часть детей уходит домой после уроков;
- оставшиеся дети из нескольких классов могут объединяться на одном обязательном допе;
- часть детей из класса может уйти на доп по выбору, а остальные остаются на обязательном допе;
- один и тот же доп по выбору может быть открыт сразу нескольким классам, но посещать его будут только записавшиеся дети;
- режимные активности вроде завтрака и обеда могут обслуживать сразу несколько классов в одном помещении.

Система должна быть пригодна для последующей трансляции своей модели данных в модель генератора расписания FET.

---

## 3. Базовые сущности предметной области

### 3.1. Класс

- Каждый ученик состоит ровно в одном классе.
- Класс является базовой "домашней" учебной группой ученика.
- Страница шаблона расписания ориентирована на строки классов.

### 3.2. Подгруппа по предмету

- Администратор может делить класс на подгруппы по отдельным предметам.
- Подгруппы используются, когда:
  - кабинет не вмещает весь класс;
  - предмет требует большей вовлеченности преподавателя;
  - в одном слоте разные части класса делают разные активности и затем могут меняться местами.
- Подгруппы не должны использоваться как механизм объединения нескольких классов на обязательных допах и режимных занятиях.

### 3.3. Группа по выбору (`ELECTIVE_GROUP`)

- Доп по выбору посещается отдельной группой по выбору.
- В группу по выбору входят только записавшиеся ученики.
- У группы по выбору есть фактический состав участников.
- У группы по выбору также есть список классов, для которых этот доп открыт.
- Состав участников и список "открытых" классов не являются одним и тем же.

### 3.4. Типы предметов

В рамках обсуждения зафиксированы следующие семантически значимые типы предметов:

- `ACADEMIC`:
  - основные учебные предметы;
  - объединение нескольких классов в одном обязательном занятии не требуется.
- `ELECTIVE_REQUIRED`:
  - обязательные дополнительные предметы;
  - предмет задается отдельно для каждого класса в требованиях;
  - при этом в шаблоне расписания несколько классов могут физически объединяться на одном занятии.
- `ELECTIVE_OPTIONAL`:
  - дополнительные предметы по выбору;
  - посещаются только записавшимися детьми;
  - могут быть открыты нескольким классам одновременно;
  - не требуют механики "объединения классов" в том же смысле, что обязательные допы, потому что их физической группой посещения уже является отдельная группа по выбору.
- `REGIME`:
  - режимные активности, например прогулка, ДЗ, завтрак, обед, полдник;
  - отдельные режимные предметы могут требовать особых правил расчета вместимости;
  - для режима также допустимо объединение нескольких целых классов в одном физическом занятии.

## 4. Ключевые инварианты

### 4.1. Инварианты по ученику

- Ученик не может физически находиться в двух местах одновременно.
- Ученик может состоять в нескольких дополнительных группах по выбору, если их занятия не пересекаются.
- Запись на доп по выбору является долгосрочной:
  - на четверть;
  - на полугодие;
  - на год;
  - но не на разовую дату.

### 4.2. Инварианты по классу

- Требования по количеству часов не объединяются между классами.
- Даже если несколько классов потом физически объединяются на одном обязательном допе, исходные требования все равно задаются отдельно по каждому классу.
- Классовая строка на странице шаблона расписания может отображать не только занятия, которые посещают все дети класса, но и все доступные сценарии для этого класса в конкретном слоте.

### 4.3. Инварианты по объединению

- Объединение допускается только для:
  - `ELECTIVE_REQUIRED`;
  - `REGIME`.
- Объединение допускается только между целыми классами.
- Подгруппы не должны участвовать в объединениях такого типа.
- Объединение может зависеть от конкретного слота:
  - классы могут быть объединены по одному предмету в понедельник;
  - и идти отдельно по тому же предмету в другой день.

### 4.4. Инварианты по отображению

- Обязательный доп должен отображаться у всего класса, даже если часть детей фактически уходит домой после уроков.
- Уход домой после продленки в текущей системе не моделируется.
- Допы по выбору должны отображаться в строках тех классов, для которых они открыты.
- Если в один слот у класса есть:
  - обязательный доп как "фоновая" активность для незаписавшихся;
  - один или несколько optional-допов;
  - режимная активность или подгруппа,
  то все такие карточки могут одновременно присутствовать в строке класса.

## 5. Требования к данным о нагрузке (`GroupSubjectRequirement`)

### 5.1. Общий принцип

Требования к нагрузке задают, сколько раз в неделю и какой длительности предмет должен быть у конкретной группы.

### 5.2. Что не меняется

- `GroupSubjectRequirement` остается отдельным для каждой группы.
- Для `ELECTIVE_REQUIRED` и `REGIME` требования по-прежнему задаются отдельно для каждого класса.
- Механика "физического объединения" нескольких классов не должна изменять семантику требований.

### 5.3. Что из этого следует

- Выполнение программы класса оценивается по его собственным требованиям.
- Если два класса объединились на одном физическом обязательном допе, это не означает, что требование существует у какой-то одной общей сущности вместо этих двух классов.

---

## 6. Требования к странице шаблона расписания

### 6.1. Роль страницы

Страница шаблона расписания является центральным местом, где администратор управляет недельным шаблоном и связанными с ним предметными сценариями.

### 6.2. Что должно настраиваться именно здесь

На странице шаблона должно быть возможно управлять:

- размещением карточек по неделе;
- обязательными допами;
- режимными предметами;
- допами по выбору;
- открытостью optional-допов для конкретных классов;
- объединением нескольких классов в одном физическом `ELECTIVE_REQUIRED` или `REGIME`-занятии.

### 6.3. Что не должно жить только в требованиях

- "Открыт для классов X/Y/Z" не является требованием нагрузки.
- "Эти два класса физически объединены на одном обязательном допе" не является требованием нагрузки.
- Эти сведения относятся к уровню шаблона расписания.

---

## 7. Правила для `ELECTIVE_OPTIONAL`

### 7.1. Смысл optional-допа

Доп по выбору:

- посещается только записавшимися детьми;
- может быть открыт нескольким классам;
- не обязан иметь тот же состав классов и участников, что другой слот того же предмета.

### 7.2. Открытость для классов

- У optional-допа должен существовать список классов, для которых он открыт.
- Этот список не зависит от фактического состава записавшихся детей.
- Optional-доп должен отображаться в строках всех классов, для которых он открыт.

### 7.3. Несколько вариантов выбора в одном слоте

Для одного и того же класса в одном и том же временном слоте может быть одновременно открыто несколько optional-допов.

Пример:

- в `16:00` для `3А` открыты и `Шахматы`, и `Театр`;
- остальные дети класса, не записавшиеся никуда, идут на обязательный доп или другую общую активность.

### 7.4. Разные открытия в разные дни

Если один и тот же предмет в разные дни открыт разным наборам классов, это должны быть разные группы по выбору, а не одна и та же группа с меняющимся списком доступности.

Пример:

- "Шахматы для 3А+3Б по понедельникам" и
- "Шахматы для 4А по средам"

должны трактоваться как две разные `ELECTIVE_GROUP`.

### 7.5. Отношение к fallback-активности

- Если ученик не записался ни на один optional-доп в конкретный слот, он может посещать обязательный доп или другую активность.
- Но система не обязана явно хранить связь "этот optional отсылает именно на этот fallback".
- Достаточно того, что в строке класса в этом слоте одновременно присутствуют все релевантные карточки.

---

## 8. Правила для `ELECTIVE_REQUIRED`

### 8.1. Смысл required-допа

Обязательный доп:

- входит в программу класса;
- должен отображаться у всех детей класса;
- не требует записи со стороны ученика;
- может физически вестись для нескольких классов сразу, если это организационно удобно школе.

### 8.2. Разделение между "положено" и "реально посещает"

- На уровне отображения обязательный доп существует у класса целиком.
- На уровне реальной посещаемости в школе возможна ситуация, когда часть детей ушла домой.
- Так как уход домой сейчас не моделируется, отображение обязательного допа у всего класса является корректным и ожидаемым поведением.

### 8.3. Объединение классов

- Несколько целых классов могут быть объединены в одном физическом обязательном допе.
- Такое объединение может меняться от слота к слоту.
- Объединение должно быть допустимым и осознанным случаем, а не считаться ошибкой конфликта.

---

## 9. Правила для `REGIME`

### 9.1. Смысл режимных активностей

Режимные активности включают, как минимум:

- ДЗ;
- прогулку;
- завтрак;
- обед;
- полдник.

### 9.2. Разные режимные предметы ведут себя по-разному

Для `REGIME` нельзя предполагать одинаковую модель загрузки помещений для всех предметов.

В обсуждении отдельно зафиксированы два класса поведения:

- питание (`Завтрак`, `Обед`, `Полдник`);
- продленочные активности (`Прогулка` и аналогичные слоты с неполной фактической явкой).

### 9.3. Допустимость совместного проведения

- Режимные активности могут проходить сразу у нескольких классов в одном помещении.
- Это особенно важно для столовой.
- Такое поведение должно поддерживаться системой как допустимый сценарий.

---

## 10. Параллельные активности в строке класса

### 10.1. Общий принцип

В один и тот же временной слот строка класса может содержать несколько карточек одновременно.

### 10.2. Допустимые причины параллельности

Параллельность допустима в следующих сценариях:

- класс разделен на подгруппы по предмету;
- часть класса идет на optional-доп, остальные остаются на обязательном допе;
- у класса одновременно открыто несколько optional-вариантов;
- часть класса находится на одной режимной активности, часть на другой, если это обеспечено подгруппами или другими доменными сущностями.

### 10.3. Что означает такая строка

Строка класса не должна интерпретироваться как "одно занятие, которое посещают все дети класса".

Она должна интерпретироваться как:

- "все занятия, которые относятся к данному классу";
- "все сценарии, которые администратор должен видеть для управления этим слотом".

---

## 11. Конфликты преподавателей и кабинетов

### 11.1. Жесткие конфликты

Для всех случаев, кроме специальных объединений `ELECTIVE_REQUIRED` и `REGIME`, пересечение по учителю или кабинету в одном слоте должно считаться ошибкой.

### 11.2. Допустимые пересечения

Один и тот же учитель может вести несколько одновременных карточек только в том случае, если это:

- осознанно объединенное занятие;
- тип предмета `ELECTIVE_REQUIRED` или `REGIME`.

То же правило действует для кабинета:

- если несколько карточек представляют один совместно проводимый обязательный доп или режимную активность, пересечение по кабинету может быть допустимым;
- во всех остальных случаях это конфликт.

### 11.3. Требование к модели данных

Модель данных должна позволять системе отличать:

- допустимое совместное проведение;
- случайную ошибку пересечения.

---

## 12. Расчет загрузки кабинета и ожидаемой численности

### 12.1. Почему это нужно

Вместимость кабинета присутствует в системе не только для справочника помещений, но и как ограничение при планировании расписания.

### 12.2. Полная численность класса

Для следующих режимных предметов нужно считать полную численность всех участвующих классов:

- `Завтрак`;
- `Обед`;
- `Полдник`.

### 12.3. Коэффициент продленки

Для следующих случаев нужно использовать оценку численности на основе коэффициента:

- `Прогулка`;
- `ELECTIVE_REQUIRED`.

Формула ожидается следующего вида:

- расчетная численность класса = `полная численность класса * глобальный коэффициент`.

### 12.4. Ограничения на ввод

- Администратор не готов вручную задавать ожидаемую численность по конкретному занятию.
- Коэффициент должен быть единым для всей школы, а не настраиваться по каждому занятию отдельно.

### 12.5. Проверка вместимости для совместных занятий

Если в одном кабинете одновременно проводится совместное `ELECTIVE_REQUIRED` или `REGIME`-занятие для нескольких классов, система должна уметь считать суммарную расчетную численность этих классов и сопоставлять ее с `seatsCount`.

---

## 13. Требования к связи с FET

### 13.1. Общий принцип

Модель данных системы должна переводиться в модель данных FET без потери доменного смысла.

### 13.2. Что FET-слой должен уметь выразить

Будущая модель должна быть способна отразить в FET:

- обычные занятия класса;
- занятия подгрупп;
- optional-занятия для отдельных групп по выбору;
- один физический required/regime-слот, который обслуживает несколько целых классов;
- правила по учителю и кабинету;
- правила вместимости с учетом полной численности или коэффициента.

### 13.3. Важное ограничение

Нельзя строить доменную модель, которая хорошо работает только в UI, но не имеет однозначного отображения в FET.

---

## 14. Нефункциональные требования к модели

### 14.1. Разделение смыслов

Модель должна явно или логически различать минимум четыре смысла:

- у кого есть требование по программе;
- кто является фактической группой посещения;
- в строках каких классов карточка отображается;
- что считается одним физическим занятием.

### 14.2. Прозрачность для админа

Администратор должен иметь возможность понять по данным и интерфейсу:

- почему карточка отображается у данного класса;
- является ли занятие общим для нескольких классов;
- является ли optional-доп только открытым, либо это обязательная активность;
- является ли пересечение допустимым или конфликтным.

### 14.3. Поддержка постепенного развития

Решение должно быть пригодно для поэтапного внедрения:

- сначала как управляемая модель шаблона;
- затем как основа для более строгой генерации и проверок.

---

## 15. Явно зафиксированные не-цели и ограничения текущего этапа

На текущем этапе система не обязана:

- моделировать фактический уход детей с продленки домой;
- вести посещаемость;
- вести журнал оценок;
- хранить ручную оценочную численность по каждому отдельному слоту;
- хранить явную связь optional-занятия с конкретным fallback-занятием.

---

## 16. Контрольные сценарии

### Сценарий A. Optional для нескольких классов

- Есть предмет `Шахматы` типа `ELECTIVE_OPTIONAL`.
- Создана группа по выбору, открытая для `3А` и `3Б`.
- В группе состоят только записавшиеся дети из этих классов.
- На странице шаблона карточка должна быть видна и в строке `3А`, и в строке `3Б`.
- При этом фактические участники optional-группы не обязаны совпадать с полной численностью обоих классов.

### Сценарий B. Несколько optional в одном слоте

- В `16:00` для `3А` открыты `Шахматы` и `Театр`.
- В том же слоте у класса есть обязательный доп для остальных детей.
- В строке `3А` должны одновременно отображаться все эти карточки.

### Сценарий C. Совместный обязательный доп

- У `3А` и `3Б` по требованиям есть обязательный доп `Архитектура`.
- В шаблоне администратор объединяет эти классы в одно физическое занятие.
- Учитель и кабинет при этом одни и те же.
- Система не должна считать такое совпадение ошибкой сама по себе.

### Сценарий D. Совместный завтрак

- Несколько классов идут в столовую в один временной слот.
- Система должна уметь проверить, что суммарная полная численность этих классов не превышает вместимость столовой.

### Сценарий E. Прогулка с коэффициентом продленки

- Несколько классов участвуют в продленочной активности `Прогулка`.
- Вместимость зоны прогулки должна проверяться не по полной численности классов, а по расчетному значению с использованием глобального коэффициента.

---

## 17. Итог

В обсуждении зафиксирован переход от простой модели "одно занятие принадлежит одной группе" к более сложной предметной области, где система обязана уметь одновременно работать с:

- классами;
- подгруппами;
- группами по выбору;
- обязательными допами;
- режимными активностями;
- карточками, отображаемыми сразу в нескольких строках классов;
- физически совместными занятиями для нескольких классов.

Любая дальнейшая модель данных для шаблона расписания должна удовлетворять этим требованиям.
