# Спецификация Версии 0.1: Фундамент и Базовый CRM

**Ответственные**: Разработчики А, Б, В

Этот документ подробно описывает интерфейсы, логику и архитектуру серверных экшенов (Server Actions) для первого этапа (Модули 1-3).

---

## 1. Архитектура роутинга (Next.js App Router)

В проекте используется группировка роутов для разделения логики авторизации и админ-панели:

- `/(auth)/login/page.tsx` — Экран входа
- `/(auth)/invite/[token]/page.tsx` — Экран активации аккаунта по инвайту
- `/(dashboard)/layout.tsx` — Общий Layout с боковым меню (Sidebar)
- `/(dashboard)/page.tsx` — Главный экран (Dashboard: статистика, кол-во учителей, учеников)
- `/(dashboard)/users/page.tsx` — Справочник пользователей (Модуль 1)
- `/(dashboard)/buildings-and-rooms/page.tsx` — Справочник помещений (Модуль 2)
- `/(dashboard)/subjects/page.tsx` — Справочник предметов (Модуль 2)
- `/(dashboard)/groups/page.tsx` — Справочник групп (Модуль 3)
- `/(dashboard)/requirements/page.tsx` — Нагрузка (Учебный план) (Модуль 3)

## 2. Модуль 1: Инструкция для Разработчика А (Инфраструктура и Пользователи)

### 2.1 Авторизация (Auth.js v5)
- **Модель базы данных**: `User`, `Account`, `Session`, `VerificationToken`
- **Домен**: Авторизация работает через `authHelpers`. Пароли не хранятся в открытом виде (используется bcrypt или argon2).
- **Инвайт-поток**:
  1. Админ вводит Имя, Фамилию и Email учителя.
  2. Server Action генерирует уникальный токен в `VerificationToken` и возвращает ссылку вида `/invite/abc123xyz`.
  3. Учитель открывает ссылку, видит свои данные, придумывает пароль.
  4. Создается Credentials запись на логин.

### 2.2 Интерфейс справочника пользователей (`/users`)
- **UI**: `TanStack Table` со столбцами (ФИО, Email, Системная роль, Доменные роли).
- **Кнопка "Добавить пользователя"**: открывает Sheet/Modal (от `shadcn/ui`).
- **Форма создания**:
  - Поля: Имя (string, min 2), Фамилия (string, min 2), Отчество (опционально), Email (z.string().email())
  - Системная Роль: `Select` (Enum: ADMIN, USER)
  - Доменная Роль (Multiple Checkboxes): Учитель, Ученик, Родитель.
- **Server Actions**: 
  - `createUserAction(data: UserFormData)`
  - `updateUserAction(id: string, data: UserFormData)`
  - `deleteUserAction(id: string)`

## 3. Модуль 2: Пространство и Дисциплины (Разработчик Б)

### 3.1 Справочник Зданий и Кабинетов (`/buildings-and-rooms`)
- **Модели базы данных**: `Building`, `Room`, `RoomSubject`
- **UI**: Разделенный экран. Слева список зданий, справа таблица кабинетов выбранного здания.
- **Интерфейс Кабинета**:
  - Название (например "Актовый зал", "Лаборатория 1")
  - Вместимость (Number)
  - **Матрица совместимости (MultiSelect)**: Выбор предметов, которые МОГУТ проходить в этом кабинете. Сохраняется в `RoomSubject`. Это критично для алгоритма FET.
- **Server Actions**:
  - `createBuildingAction(data)`, `updateBuildingAction(id, data)`, `deleteBuildingAction(id)`
  - `createRoomAction(buildingId, data)`, `updateRoomAction(id, data)`, `deleteRoomAction(id)`

### 3.2 Справочник Предметов (`/subjects`)
- **Модели базы данных**: `Subject`, `SubjectType` Enum
- **UI**: Простая `TanStack Table` или список (Card grid).
- **Форма предмета**:
  - Название (string)
  - Тип: Радиокнопки (`ACADEMIC`, `ELECTIVE`, `REGIME`).
  - *Важно: предметы с типом `REGIME` (например, "Обед") будут по-особому обрабатываться FET-генератором.*
- **Server Actions**: `createSubjectAction(...)`, `updateSubjectAction(...)`, `deleteSubjectAction(...)`

## 4. Модуль 3: Люди и Нагрузка (Разработчик В)

### 4.1 Справочник Групп (`/groups`)
- **Модели базы данных**: `Group`, `GroupType` Enum, `StudentGroups`
- **UI**: Деревовидная таблица (Tree Table) или список аккордеонов.
- **Структура**: Классы (`CLASS`) содержат внутри себя Подгруппы (`SUBJECT_SUBGROUP`). Дополнительные/Сборные группы (`ELECTIVE_GROUP`) живут на верхнем уровне.
- **Форма создания группы**:
  - Название ("2А", "Английский 2А-1", "Робототехника")
  - Тип (`Select`)
  - Класс (grade: 1-11 или NULL). Для садика = NULL.
  - Состав (привязка конкретных Student). Открывает модалку с `MultiSelect` из учеников.
- **Server Actions**: `createGroupAction(...)`, `assignStudentsToGroupAction(groupId, studentIds[])`.

### 4.2 Компетенции учителей (Часть карточки Учителя в `/users`)
- **Модель**: `TeacherSubject`
- **Интерфейс**: В профиле учителя (после создания) появляется таб "Компетенции".
- Учителю назначаются `Subject` с указанием `minGrade` и `maxGrade` (например: Математика, классы с 5 по 7).

### 4.3 Учебный план (Супер-интерфейс) (`/requirements`)
- **Модель базы данных**: `GroupSubjectRequirement`
- Данный экран — **сердце CRM**. 
- **UI**: Огромная Data Table (TanStack Table). Строки — Группы, Столбцы — Предметы.
- Этот интерфейс должен быть написан с использованием `TanStack Form` для **Inline Editing**.
- **Ячейка таблицы**:
  - Количество уроков в неделю (Input Number)
  - Длительность урока (Input Number, в минутах)
  - Нужна ли большая перемена (Input Number, в минутах, default: 0)
- **Сохранение**: При потере фокуса (`onBlur`) или нажатии "Сохранить строку" вызывается `upsertRequirementAction(groupId, subjectId, data)`.
