# Спецификация Версии 0.1: Фундамент и Базовый CRM

Этот документ описывает техническую архитектуру, маппинг роутинга и слой базы данных для первой фазы проекта, синхронизируя серверную логику с утвержденными UI/UX спецификациями (Справочники, Пользователи, Кабинеты).

---

## 1. Архитектура роутинга (Next.js App Router)

В проекте используется группировка роутов (Route Groups) для разделения логики авторизации и защищенного дашборда:

- `(auth)/login/page.tsx` — Экран входа в систему.
- `(auth)/invite/[token]/page.tsx` — Экран активации аккаунта и задания пароля по пригласительной ссылке.
- `(dashboard)/layout.tsx` — Глобальный **Side Navigation Layout** (Сворачиваемый Sidebar без верхнего Header'а).
- `(dashboard)/page.tsx` — Главный экран (Сводная статистика).
- `(dashboard)/users/page.tsx` — Справочник пользователей (Конвейерный табличный ввод).
- `(dashboard)/rooms/page.tsx` — Справочник зданий и кабинетов (Master-Detail вид).
- `(dashboard)/subjects/page.tsx` — Справочник предметов.
- `(dashboard)/groups/page.tsx` — Справочник групп (Tree Table).
- `(dashboard)/requirements/page.tsx` — Матрица учебного плана (Супер-интерфейс).

## 2. Модуль: Пользователи и Авторизация (`/users`)

### 2.1 Подсистема Авторизации (Auth.js v5)
- **Модель базы данных**: `User`, `Account`, `Session`, `VerificationToken`.
  А также доменные роли: `Teacher`, `Student`, `Parent`. Один `User` может иметь несколько доменных ролей.
- **Инвайт-поток (Активация)**:
  1. Администратор создает аккаунт (без пароля), Server Action генерирует `VerificationToken` и выдает код/ссылку вида `/invite/abc123xyz`.
  2. Пользователь открывает ссылку. Система проверяет токен.
  3. Пользователь видит форму завершения регистрации: сам вводит или исправляет свой Email, ФИО и придумывает Пароль.
  4. Создается Credentials запись на логин, статус меняется на `ACTIVE`.

### 2.2 Интерфейс справочника пользователей
- **UI**: Data Grid (TanStack Table) на всю ширину экрана.
- **Создание пользователей (Smart Row / Inline Edit)**:
  - Форма вынесена в саму таблицу. По кнопке "Добавить" появляется новая интерактивная строка в топе таблицы.
  - Администратор работает клавиатурой (Mouse-free UX): ФИО -> `Tab` -> Email (Опционально) -> `Tab` -> Роль -> `Enter`.
  - При сохранении строка фиксируется, и тут же под ней появляется новая пустая строка для конвейерного ввода целого класса.
  - Если Email не был введен, в колонке действий появляется кнопка генерации кода-ссылки.
- **Работа с Родителями (Однокликовый инвайт)**:
  - Родители не добавляются через общую форму. Завуч кликает `...` на конкретном ученике и выбирает `[Сгенерировать инвайт для родителя]`, получая код для WhatsApp, ИЛИ связывает ученика с уже существующим родителем (брат/сестра).
- **Server Actions**: 
  - `createUserAction(data)`, `generateParentInviteAction(studentId)`
  - `updateUserAction(id, data)` (Вызов боковой панели - Sheet для кросс-ролевого назначения, например дать Учителю права Родителя).
  - `deleteUserAction(id)`

## 3. Модуль: Инфраструктура и Дисциплины

### 3.1 Справочник Зданий и Кабинетов (`/rooms`)
- **Модели базы данных**: `Building`, `Room`, `RoomSubject`.
- **UI**: Разделенный экран (Master-Detail). Слева навигация по зданиям, справа таблица кабинетов выбранного здания.
- **Интерфейс Кабинета (Двухшаговое создание)**:
  - Шаг 1: Простое создание Кабинета (Modal Dialog) — название и вместимость.
  - Шаг 2: Привязка предметов (Tactile Buckets). Широкий интерфейс с двумя колонками ("Доступные предметы" и "Предметы кабинета"). Администратор распределяет дисциплины с помощью интуитивного Drag-and-Drop (`dnd-kit`). Сохраняется в БД через механизм Optimistic UI.
- **Server Actions**:
  - `createBuildingAction(data)`, `updateBuildingAction(id, data)`, `deleteBuildingAction(id)`
  - `createRoomAction(buildingId, data)`, `updateRoomSubjectsAction(roomId, subjectIds[])`

### 3.2 Справочник Предметов (`/subjects`)
- **Модель базы данных**: `Subject` (Enum типа предмета: `ACADEMIC`, `ELECTIVE`, `REGIME`).
- **UI**: Компактная `TanStack Table`, сгруппированная по типам предметов. Создание происходит через простой `Dialog`.

## 4. Модуль: Люди и Нагрузка (Группы и План)

### 4.1 Справочник Групп (`/groups`)
- **Модели базы данных**: `Group`, `GroupType` Enum, `StudentGroups`
- **UI**: Деревовидная таблица (Tree Table) или список аккордеонов.
- **Структура**: Классы (`CLASS`) содержат внутри себя Подгруппы (`SUBJECT_SUBGROUP`). Дополнительные/Сборные группы (`ELECTIVE_GROUP`) живут на верхнем уровне.
- **Форма создания и наполнения**:
  - Ученики набираются в классы и кружки через **Transfer List (Dual Listbox)** с фильтрами по классам.
  - Разделение базового класса на подгруппы (например, английский язык) происходит через **Splitter UI**: интерактивный канбан-интерфейс для перетаскивания учеников из общего пула в колонки подгрупп или с кнопкой автоматического равного деления.
- **Server Actions**: `createGroupAction(...)`, `assignStudentsToGroupAction(groupId, studentIds[])`.

### 4.2 Учебный план (Супер-интерфейс) (`/requirements`)
- **Модель базы данных**: `GroupSubjectRequirement`
- Данный экран — **сердце CRM**. 
- **UI**: Огромная Data Table (TanStack Table) на весь экран (отступы убраны). Строки — Группы, Столбцы — Предметы.
- Этот интерфейс должен быть написан с использованием `TanStack Form` для **Inline Editing**.
- **Ячейка таблицы**:
  - Количество уроков в неделю (Input Number)
  - Длительность урока (Input Number, в минутах)
  - Нужна ли большая перемена (Input Number, в минутах, default: 0)
- **Сохранение**: При потере фокуса (`onBlur`) или нажатии "Сохранить строку" (Enter) вызывается `upsertRequirementAction(groupId, subjectId, data)` с оптимистичным обновлением UI (Optimistic Updates).

## 5. Ожидаемые UI Компоненты от `shadcn/ui`
Для реализации требований v0.1 необходимо перенести в проект следующие компоненты:
1. `Button`, `Input`, `Label`.
2. `Card`, `Sheet` (Выезжающая панель), `Dialog` (Модалки).
3. `Table` (Базовый рендерер для TanStack).
4. `DropdownMenu`, `Popover`, `Select`, `Checkbox`, `Radio Group`.
5. `Form` (Для простых форм).
6. `Sidebar` (Навигация).
7. `Sonner` (Глобальный Toaster для уведомлений об успехе/ошибке Server Actions).
