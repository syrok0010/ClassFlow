# Спецификация UI/UX: Доступность преподавателя (`/teacher/availability`)

Teacher-side экран для заполнения собственной доступности.

Он работает с двумя слоями данных:
- **`TeacherAvailability`**: регулярный недельный шаблон;
- **`TeacherAvailabilityOverride`**: разовые исключения на конкретные даты.

---

## 1. Модель и правила

Связанные сущности Prisma:
- **`TeacherAvailability`**: `teacherId`, `dayOfWeek`, `startTime`, `endTime`, `type`
- **`TeacherAvailabilityOverride`**: `teacherId`, `startTime`, `endTime`, `type`

Enum `AvailabilityType`:
- `PREFERRED`
- `AVAILABLE`
- `UNAVAILABLE`

Правила:
- `TeacherAvailability` задаёт повторяющийся недельный шаблон;
- `TeacherAvailabilityOverride` переопределяет его на конкретные даты;
- teacher-side экран является основным местом ввода этих данных;
- администратор использует отдельный availability-интерфейс для анализа и точечной правки.

---

## 2. Роль экрана

Маршрут:
- `/teacher/availability`

Вход:
- sidebar -> `Мое преподавание` -> `Моя доступность`

Что должен уметь преподаватель:
- указать регулярный график;
- отметить предпочтительные слоты;
- добавить временное исключение;
- увидеть, как override меняет текущую неделю.

---

## 3. Layout

Экран строится по паттерну **preview-first**.

### Header
- `H1`: `Моя доступность`
- подзаголовок:
  - `Укажите ваш обычный график и разовые изменения. Завуч использует эти данные при составлении расписания и поиске замен.`

### Основная область
1. **Превью текущей недели**
- главный и самый крупный блок страницы;
- по умолчанию открывается текущая неделя;
- показывает итог недели с учётом:
  - регулярного шаблона;
  - override.

2. **Боковая колонка override**
- список активных и будущих исключений;
- кнопка `[+ Добавить исключение]`;
- быстрый доступ к просмотру, удалению и созданию override.

---

## 4. Визуальные состояния

На сетке используются четыре состояния:

1. **Пусто**
- ничего не указано

2. **Предпочтительно** (`PREFERRED`)
- насыщенный зелёный

3. **Доступно** (`AVAILABLE`)
- светлый зелёный

4. **Недоступно** (`UNAVAILABLE`)
- серый или красноватый тон

Override показывается поверх базового шаблона:
- полупрозрачным блоком;
- с явной границей, паттерном или tooltip `Исключение`.

Пользователь должен видеть и базовый шаблон, и то, что именно изменил override.

---

## 5. Регулярный шаблон

Регулярный шаблон редактирует `TeacherAvailability`.

### Сетка
- по оси Y: `Пн-Пт`
- по оси X: рабочее время, например `08:00-18:00`

### Действия на сетке
При hover на день или слот появляются две плавающие кнопки:
- `Добавить доступность`
- `Добавить исключение`

`Добавить доступность`:
- работает с регулярным шаблоном;
- открывает inline-действие для создания или изменения интервала;
- после этого пользователь выбирает тип:
  - `Предпочтительно`
  - `Доступно`
  - `Недоступно`
  - `Стереть`

Поведение:
- интервалы одного типа должны схлопываться;
- пересечения должны нормализоваться;
- пустые зоны не создают запись в БД.

Подсказка:
- `Отмечайте, когда вы обычно можете работать каждую неделю.`

---

## 6. Override и исключения

Override редактируют `TeacherAvailabilityOverride`.

### Основной смысл
Дать преподавателю быстрый способ сказать:
- `в эту дату я недоступен`;
- `в этот день у меня появилось дополнительное окно`;
- `в этот день этот слот предпочтителен`.

### Колонка override
Каждая запись показывает:
- диапазон даты и времени;
- тип (`PREFERRED`, `AVAILABLE`, `UNAVAILABLE`);
- действие удаления.

При клике на запись соответствующий блок на превью недели подсвечивается.

### Создание override
Создаётся:
- по кнопке `[+ Добавить исключение]` в боковой колонке;
- или по кнопке `Добавить исключение` на сетке.

Поля:
- `Дата / дата-диапазон`
- `Время начала`
- `Время окончания`
- `Тип`

Тексты типов:
- `Предпочтительно`
- `Доступно`
- `Недоступно`

Поведение:
- `Enter` сохраняет;
- `Esc` отменяет;
- удаление по `Trash` + `AlertDialog`.

Валидация:
- дата обязательна;
- `startTime < endTime`;
- override не может иметь нулевую длину.

---

## 7. Превью недели

Превью показывает итог недели:
- сначала регулярный шаблон;
- затем override поверх него.

Зачем оно нужно:
- без него override остаются абстрактным списком дат;
- с ним преподаватель сразу видит, как исключение меняет неделю.

Управление:
- селектор недели:
  - `< 15–21 сентября >`
- по умолчанию открывается текущая неделя.

---

## 8. Empty states

### Нет регулярного шаблона
- `Вы еще не указали регулярную доступность`
- подпись:
  - `Начните с превью недели: добавьте обычные интервалы доступности или первое исключение`

### Нет override
- `У вас нет разовых исключений`
- подпись:
  - `Добавляйте исключения для больничных, отпусков, переносов и дополнительных окон`

### Пустая неделя
- `На этой неделе у вас пока нет отмеченной доступности`

---

## 9. Actions и доступ

Минимально ожидаемые actions:
- `getTeacherAvailabilityAction({ teacherId?, weekStart? })`
- `upsertTeacherAvailabilityAction({ teacherId?, entries })`
- `createTeacherAvailabilityOverrideAction({ teacherId?, startTime, endTime, type })`
- `updateTeacherAvailabilityOverrideAction({ teacherId?, overrideId, startTime, endTime, type })`
- `deleteTeacherAvailabilityOverrideAction({ teacherId?, overrideId })`

### Аутентификация
Каждый action должен проверять:
- у запроса есть валидная сессия;
- пользователь аутентифицирован.

### Авторизация
Teacher-side:
- преподаватель может читать и изменять только свою доступность;
- `teacherId` не должен доверяться клиенту и берётся из сессии;
- если пользователь не имеет доменной роли `Teacher`, action возвращает ошибку авторизации.

Admin-side:
- `ADMIN` может читать и изменять availability любого преподавателя;
- `teacherId` в admin-контексте может приходить из маршрута или параметров.

---

## 10. Резюме для разработчика

- Это teacher-side экран `/teacher/availability`.
- Основной паттерн: текущая неделя по центру, override в боковой колонке.
- `Добавить доступность` работает с регулярным шаблоном.
- `Добавить исключение` создаёт override.
- Override должны быть видны и в колонке, и на превью недели.
