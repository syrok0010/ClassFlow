# Спецификация UI/UX: Управление предметами (`/admin/subjects`)

Этот документ описывает интерфейс страницы администрирования предметов. Экран нужен для поддержки базового справочника `Subject`, на который затем опираются кабинеты, учебный план, компетенции учителей и генерация расписания.

---

## 1. Контекст и роль страницы

Оперируемая сущность БД:
- **`Subject`**: `id`, `name`, `type`.

Enum `SubjectType`:
- **`ACADEMIC`**: основная академическая программа.
- **`ELECTIVE`**: дополнительные занятия, кружки и факультативы.
- **`REGIME`**: режимные моменты, которые участвуют в расписании как обычные активности (`обед`, `прогулка`, `сон`, `тьюторский блок`).

Экран является **единым справочником активностей школы**, а не просто списком "школьных дисциплин". Это критично для всей доменной модели:
- в [`ui-rooms.md`](./ui-rooms.md) предметы привязываются к кабинетам через `RoomSubject`;
- в [`ui-requirements.md`](./ui-requirements.md) предметы становятся колонками матрицы нагрузки;
- в `TeacherSubject` эти же предметы задают компетенции преподавателей;
- в генераторе расписания `REGIME`-сущности участвуют наравне с уроками.

Из этого следуют два правила:
1. Нельзя делать страницу слишком "академической" по терминологии.
2. Нельзя позволять бесконтрольное удаление предметов, если на них уже завязаны другие модули.

## 2. Макет страницы (Layout)

Страница следует паттерну уже существующих административных экранов: заголовок, краткое описание, toolbar и компактная табличная область без сложного split-view.

### Header (Шапка)
- Заголовок `H1`: "Предметы".
- Подзаголовок: "Управление списком предметов и режимных моментов."
- Основная кнопка справа: `[+ Добавить предмет]`.

### Toolbar (Панель управления)
- `Поиск` по названию.
- `Фильтр по типу`: `Все`, `ACADEMIC`, `ELECTIVE`, `REGIME`.
- `Сортировка`: по названию или по типу.
- Опциональный переключатель `Группировать по типу`.

### Основной контент
- Основной вид: **компактная `TanStack Table`**.
- По умолчанию таблица сгруппирована по типам предметов, как и указано в `v0.1-foundation.md`.
- Каждая группа имеет свой визуальный заголовок:
  - `Основная программа`
  - `Дополнительные занятия`
  - `Режимные моменты`

### Колонки таблицы
1. `Название`
2. `Тип`
3. `Где используется`
4. `Действия`

Содержимое колонок:
- **`Название`**: имя предмета, основной акцент строки.
- **`Тип`**: цветной badge.
- **`Где используется`**: компактные счетчики-связи:
  - `Кабинеты: 3`
  - `Требования: 12`
  - `Учителя: 4`
  Если связей пока нет, показывается `Не используется`.
- **`Действия`**: меню `...` -> `Удалить`.

## 3. Empty States (Пустые состояния)

### А. В системе нет ни одного предмета
- Центрированный empty state внутри карточки.
- Текст: "Справочник предметов пока пуст".
- Подпись: "Добавьте академические предметы, допы и режимные моменты, чтобы настроить кабинеты и учебный план."
- Кнопка: `[+ Добавить первый предмет]`.

### Б. Поиск или фильтры ничего не нашли
- Иконка поиска.
- Текст: "Ничего не найдено".
- Кнопка-ссылка: `Сбросить фильтры`.

## 4. Создание предмета (Inline Create Flow)

Для этой страницы уместнее **inline-создание в таблице**, а не модальный `Dialog`. Причина простая: справочник предметов часто заводится сериями, а сама сущность компактная и не требует много полей. По UX это ближе к `Smart Row`, чем к отдельной форме.

### Механика
1. Администратор нажимает `[+ Добавить предмет]`.
2. В верхней части таблицы появляется интерактивная строка создания.
3. Фокус сразу ставится в поле `Название`.
4. Дальше клавиатурный поток:
   - `Название` -> `Tab`
   - `Тип` -> `Enter`

### Поля inline-строки
1. `Название`
   - обязательное;
   - обрезается по краям (`trim`);
   - уникально в рамках справочника;
   - нельзя сохранить имя, уже существующее в любой другой строке.
2. `Тип`
   - `ACADEMIC`
   - `ELECTIVE`
   - `REGIME`

### Поведение после сохранения
- Строка уходит в обычный режим чтения.
- Новая запись появляется в нужной секции таблицы.
- Сразу создается новая пустая inline-строка, чтобы можно было продолжать потоковый ввод.
- Выход из режима:
  - `Esc` отменяет создание;
  - клик вне строки отменяет пустую строку;
  - если введены данные, но есть ошибка валидации, строка не исчезает.

### Валидация
- Нельзя сохранить пустое название.
- Нельзя сохранить предмет с именем, уже существующим в системе.
- Ошибка показывается прямо в строке, под полем названия: `Предмет с таким названием уже существует`.

## 5. Редактирование предмета

Отдельный диалог `Редактировать` для этой страницы не нужен.

### Переименование
- Переименование запускается **двойным кликом по названию** или действием `Enter` на сфокусированной ячейке.
- Ячейка `Название` переходит в inline-режим редактирования.
- `Enter` сохраняет.
- `Esc` отменяет.
- Валидация та же, что и при создании:
  - пустое имя запрещено;
  - имя должно быть уникальным.

### Изменение типа
Смена типа предмета **запрещена**.

Причина: это затрагивает:
- визуальную группировку списка;
- положение предмета в матрице учебного плана;
- смысл связей с кабинетами;
- будущие ограничения генератора расписания;
- доменную корректность `REGIME`-сущностей.

Итоговое правило:
- inline-редактирование имени разрешено;
- поле `type` доступно только при создании;
- после создания тип нигде в UI не редактируется;
- `updateSubjectAction` не должен принимать изменение `type`, даже если клиент попытается его отправить вручную.

## 6. Удаление и защита от дурака

Удаление запускается из `... -> Удалить` и всегда требует `AlertDialog`.

### Сценарий 1: Предмет нигде не используется
- Показывается обычное подтверждение:
  - "Удалить предмет `Робототехника`?"
  - "Это действие нельзя отменить."

### Сценарий 2: Предмет уже связан с другими сущностями
Удаление **запрещено**, если предмет используется хотя бы в одном месте:
- есть записи в `RoomSubject`;
- есть записи в `GroupSubjectRequirement`;
- есть записи в `TeacherSubject`;
- в будущем может появиться связь с `WeeklyScheduleTemplate` или `ScheduleEntry`.

В этом случае:
- кнопка удаления в меню может оставаться доступной, но вести в защищенный диалог;
- диалог показывает причины блокировки:
  - `Используется в 2 кабинетах`
  - `Используется в 5 требованиях учебного плана`
  - `Назначен 3 преподавателям`
- финальное действие `Удалить` disabled.

Формулировка ошибки:
- "Невозможно удалить предмет, пока он используется в других разделах системы."

## 7. Связи с другими модулями

Эта страница должна явно показывать, что предметы не живут изолированно.

Отдельно: отображение связей здесь нужно, но редактирование этих связей из этой же таблицы может перегрузить интерфейс и смешать ответственность модулей. Поэтому в спецификации фиксируем только базовый безопасный уровень.

### 7.1. Интеграция с кабинетами (`/admin/rooms`)
- После создания предмет сразу должен быть доступен в интерфейсе сопоставления кабинетов.
- Для `REGIME`-предметов особенно важны пространства вроде столовой, площадки, спортзала, купола.

### 7.2. Интеграция с учебным планом (`/admin/requirements`)
- Справочник предметов определяет набор колонок матрицы нагрузки.
- Удаление или переименование предмета должно аккуратно отражаться на матрице.

### 7.3. Интеграция с будущими компетенциями учителей
- Предметы должны быть пригодны не только для расписания, но и для `TeacherSubject`.
- Поэтому названия должны быть нормализованы и без дублей вроде `Математика`, `математика`, `МАТЕМАТИКА`.

### 7.4. Интеграция с генерацией расписания
- `REGIME`-предметы не являются второстепенными. Для системы это такие же сущности планирования, как и уроки.
- Поэтому режимные моменты должны визуально быть "полноправными", а не спрятанными в отдельный служебный справочник.

### 7.5. Как показывать связи в этом View
В базовой версии колонка `Где используется` остается обзорной, а не управляющей.

Что показываем:
- счетчики по основным направлениям:
  - `Кабинеты`
  - `Требования`
  - `Учителя`
- по клику на счетчик можно открыть popover, tooltip или sheet с деталями:
  - список кабинетов;
  - список групп, где предмет фигурирует в требованиях;
  - список учителей, у которых есть эта компетенция.

Что **не** делаем в этом экране в первой версии:
- не редактируем `RoomSubject`;
- не редактируем `GroupSubjectRequirement`;
- не редактируем `TeacherSubject`.

Причина:
- эти связи имеют собственные специализированные интерфейсы;
- встроенное редактирование из списка предметов создаст перегруженный "экран-всё-в-одном";
- риск случайно сломать связанную конфигурацию здесь слишком высокий.

Итоговое решение для спецификации:
- в этом View связи **просматриваются**;
- управление связями происходит в профильных модулях;
- позже можно добавить deep links:
  - `Открыть в кабинетах`
  - `Открыть в учебном плане`
  - `Открыть у преподавателей`

## 8. Визуальная семантика

Чтобы экран быстро считывался, у каждого типа предмета должен быть свой устойчивый визуальный код:
- `ACADEMIC`: спокойный синий или индиго.
- `ELECTIVE`: теплый оранжевый.
- `REGIME`: нейтральный зеленый или бирюзовый.

Эти же цвета желательно переиспользовать:
- в badge на странице предметов;
- в матрице учебного плана;
- в редакторе кабинетов;
- позже в расписании и генераторе.

Цель: чтобы администратор по цвету мгновенно понимал, что перед ним академическая нагрузка, доп или режимный блок.

## 9. Server Actions и требования к данным

Минимально ожидаемые операции:
- `getSubjectsAction(filters)`
- `createSubjectAction(data)`
- `updateSubjectAction(id, data)`
- `deleteSubjectAction(id)`

Дополнительно желательно:
- `getSubjectUsageAction(id)` или включение usage-статистики в основной список.

Валидация:
- `name.trim()` обязателен;
- дубли имен не допускаются;
- проверка уникальности обязательна и при обычном создании, и при inline-переименовании;
- при удалении сервер обязан повторно проверять связи, даже если клиент уже показал блокировку.

---

## 10. Резюме для разработчика

- Это не просто CRUD-таблица, а опорный справочник для всей доменной модели.
- UX должен быть проще, чем у `/admin/users` и `/admin/requirements`: компактная таблица с inline-созданием и inline-переименованием.
- Главное отличие от "обычного списка предметов" в том, что `REGIME`-активности являются равноправными сущностями и должны быть видимы в общем потоке.
- Главный риск экрана не создание, а корректное управление типом и безопасное удаление связанного предмета.
