# Спецификация UI/UX: Компетенции преподавателя (`TeacherSubject`)

Этот документ описывает один и тот же набор данных `TeacherSubject` в двух интерфейсах:
- teacher-side экран, где преподаватель сам заполняет свои предметы;
- admin-side экран, где администратор просматривает и при необходимости точечно правит эти данные.

Документ общий, потому что доменная модель и почти весь CRUD у этих экранов одинаковые. Разница в основном в точке входа и в контексте использования.

---

## 1. Модель и правила

Оперируемые сущности:
- **`Teacher`**: доменная роль преподавателя.
- **`Subject`**: предмет или активность.
- **`TeacherSubject`**: связь преподавателя с предметом и диапазоном классов.

Поля `TeacherSubject`:
- `teacherId`
- `subjectId`
- `minGrade`
- `maxGrade`

Принятые правила:
- один преподаватель по одному предмету имеет один **сплошной** диапазон классов;
- `REGIME`-предметы тоже допустимы;
- один `TeacherSubject` на пару `teacherId + subjectId`;
- изменение предмета у существующей записи **запрещено**;
- если предмет выбран неверно, запись удаляется и создается заново.

Сервер обязан поддерживать те же ограничения независимо от UI.

---

## 2. Общий паттерн интерфейса

Оба экрана используют один и тот же базовый паттерн:
- компактная таблица или плотный список строк;
- inline-создание новой записи;
- inline-редактирование только `minGrade` и `maxGrade`;
- без `Dialog` для создания и изменения записи;
- удаление через иконку `Trash` с подтверждением в `AlertDialog`.

Базовые колонки:
1. `Предмет`
2. `Тип`
3. `От класса`
4. `До класса`
5. `Действия`

Поведение строки:
- `Предмет` и `Тип` только читаются;
- клик по `От класса` редактирует `minGrade`;
- клик по `До класса` редактирует `maxGrade`;
- `Enter` сохраняет;
- `Esc` отменяет;
- предмет у существующей строки не меняется.

Создание:
- кнопка `[+ Добавить предмет]` добавляет inline-строку в начало списка;
- поля новой строки:
  - `Предмет`
  - `От класса`
  - `До класса`
- валидация:
  - предмет обязателен;
  - `minGrade <= maxGrade`;
  - нельзя создать дубликат предмета для одного преподавателя.

Удаление:
- запускается по иконке `Trash`;
- подтверждение через `AlertDialog`;
- используется в том числе как официальный способ исправить ошибочно выбранный предмет.

## 3. Teacher View

### Маршрут и вход
- маршрут: `/teacher/subjects`
- вход: sidebar -> `Мое преподавание` -> `Мои предметы`

### Роль экрана
Это основной экран ввода `TeacherSubject`.

Главное пользовательское намерение:
- заполнить свои предметы;
- указать диапазоны классов;
- поправить ошибку без обращения к администратору.

### Особенности UI
- экран **edit-first**;
- под заголовком показывается короткая поясняющая подсказка:
  - `Завуч использует этот список при составлении расписания и поиске замен.`
  - `Если вы выбрали не тот предмет, удалите запись и создайте новую.`

### Header
- `H1`: `Мои предметы`
- подзаголовок:
  - `Укажите, какие предметы и для каких классов вы ведете.`

### Empty state
- `У вас пока не добавлено ни одного предмета`
- подпись:
  - `Добавьте предметы и диапазоны классов, чтобы завуч мог учитывать вас при составлении расписания`

---

## 4. Admin View

### Маршрут и вход
- маршрут: `/admin/users/[id]/teaching`
- вход: `/admin/users` -> меню действий пользователя-учителя -> `Компетенции`

### Роль экрана
Это экран просмотра и точечной корректировки данных конкретного преподавателя.

Главное пользовательское намерение:
- быстро понять, что ведет конкретный человек;
- при необходимости поправить диапазон;
- при ошибке удалить запись и создать правильную.

### Особенности UI
- экран **read-first**, но без отдельного режима редактирования;
- header содержит контекст преподавателя, а не только список его предметов.

### Header
- breadcrumbs:
  - `Пользователи / Имя преподавателя / Компетенции`
- `H1`: `Компетенции преподавателя`
- вторичная информация:
  - ФИО
  - email
  - роли / статус

### Empty state
- `У преподавателя пока не назначено ни одного предмета`
- подпись:
  - `Добавьте предметы и диапазоны классов, чтобы система могла учитывать этого преподавателя в учебном плане и расписании`

---

## 5. Фильтры и поиск

Для обеих версий используются:
- поиск по названию предмета;
- фильтр по типу:
  - `Все`
  - `ACADEMIC`
  - `ELECTIVE`
  - `REGIME`

Если фильтр или поиск ничего не нашли:
- `Ничего не найдено`
- действие:
  - `Сбросить фильтры`

## 6. Server Actions и данные

Для teacher-side и admin-side используется один и тот же набор actions:
- `getTeacherSubjectsAction({ teacherId?, filters? })`
- `createTeacherSubjectAction({ teacherId?, subjectId, minGrade, maxGrade })`
- `updateTeacherSubjectAction({ teacherId?, subjectId, minGrade, maxGrade })`
- `deleteTeacherSubjectAction({ teacherId?, subjectId })`

Требования:
- `teacherId` должен определяться из контекста:
  - в teacher-side интерфейсе берется из текущей сессии;
  - в admin-side интерфейсе берется из маршрута или параметров;
- предмет должен существовать;
- `minGrade` и `maxGrade` должны быть валидными числами;
- `minGrade <= maxGrade`;
- нельзя создать дубликат пары `teacherId + subjectId`;
- update-операция не должна позволять менять `subjectId`.

### Аутентификация
Каждый action должен проверять, что:
- у запроса есть валидная сессия;
- пользователь аутентифицирован.

### Авторизация
Каждый action должен проверять, что:
- пользователь является `ADMIN`, либо
- пользователь имеет доменную роль `Teacher`.

Дополнительные правила доступа:
- `ADMIN` может читать и изменять `TeacherSubject` для любого преподавателя;
- преподаватель может читать и изменять только собственные `TeacherSubject`;
- если teacher-side запрос пытается передать чужой `teacherId`, action должен игнорировать его и брать `teacherId` из сессии;
- если пользователь не является `ADMIN` и не имеет роли `Teacher`, action должен возвращать ошибку авторизации;
- если запрошенный `teacherId` не существует или не принадлежит доступному контексту, action должен возвращать ошибку доступа или отсутствия сущности.

---

## 7. Резюме для разработчика

- Это одна общая спецификация для двух экранов с одной моделью данных.
- Разница между ними не в CRUD, а в контексте:
  - teacher-side экран является основным местом ввода;
  - admin-side экран нужен для просмотра и точечной правки.
- В обеих версиях:
  - создание inline;
  - редактирование только диапазона классов;
  - предмет в существующей записи не меняется;
  - исправление предмета делается через удаление и создание новой записи.
