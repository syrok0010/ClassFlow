# Спецификация UI/UX: Компетенции преподавателя для администратора

Этот документ описывает административный интерфейс просмотра и точечной правки `TeacherSubject` для конкретного преподавателя.

Экран нужен администратору в первую очередь для **обзора**: что ведёт конкретный учитель, для каких возрастов и где это уже используется в системе. При этом интерфейс должен позволять вносить редкие корректировки, не превращаясь в тяжёлую форму и не прячась в `Sheet` профиля пользователя.

---

## 1. Контекст и роль страницы

Оперируемые сущности:
- **`User`**: учетная запись человека.
- **`Teacher`**: доменная роль преподавателя.
- **`Subject`**: предмет или активность.
- **`TeacherSubject`**: связь преподавателя с предметом и диапазоном классов.

Поля `TeacherSubject`:
- `teacherId`
- `subjectId`
- `minGrade`
- `maxGrade`

Принятые правила для первой версии:
- один преподаватель по одному предмету имеет один **сплошной** диапазон классов;
- `REGIME`-предметы тоже допустимы как компетенции преподавателя;
- основной сценарий заполнения этих данных принадлежит учителю в личном кабинете;
- административный экран нужен для просмотра и точечной корректировки;
- изменение предмета у существующей записи **запрещено**:
  если выбран не тот предмет, администратор удаляет связь и создаёт новую.

---

## 2. Где находится экран

Экран открывается **со страницы пользователей**.

Сценарий:
1. Администратор открывает `/admin/users`.
2. Находит пользователя с ролью учителя.
3. В меню действий `...` выбирает пункт:
   - `Компетенции`
   или
   - `Предметы преподавателя`

Результат:
- происходит переход на отдельную страницу конкретного преподавателя;
- экран не живёт внутри `Sheet` редактирования пользователя.

Зафиксированный маршрут:
- `/admin/users/[id]/teaching`

---

## 3. Общий паттерн интерфейса

Экран строится как **read-first страница со встроенным inline-редактированием отдельных строк**.

Принципы:
- без глобального режима `Просмотр / Редактирование`;
- без `Dialog` для добавления и изменения записи;
- без редактирования предмета у существующей строки;
- редактирование локально и только там, где это действительно нужно.

Это означает:
- администратор по умолчанию видит список компетенций;
- кнопка `+ Добавить предмет` создаёт inline-строку;
- кнопка `Изменить` у строки переводит в редактирование только диапазон классов;
- если нужно поменять предмет, используется сценарий `Удалить` + `Создать`.

## 4. Макет страницы (Layout)

### Header
- Breadcrumbs:
  - `Пользователи / Мария Иванова / Компетенции`
- Заголовок `H1`: `Компетенции преподавателя`
- Подзаголовок:
  - ФИО преподавателя
  - вторичная информация: email, роли, статус аккаунта

### Summary-блок
Компактная строка сводки под заголовком:
- `Всего предметов: 6`
- `Academic: 4`
- `Elective: 1`
- `Regime: 1`
- `Диапазон покрытия: 0-7 классы`

Этот блок нужен именно администратору, потому что он помогает быстро понять профиль человека до чтения таблицы.

### Toolbar
- Кнопка: `[+ Добавить предмет]`
- Поиск по названию предмета
- Фильтр по типу:
  - `Все`
  - `ACADEMIC`
  - `ELECTIVE`
  - `REGIME`

### Основной контент
Компактная таблица или плотный список строк.

Рекомендуемые колонки:
1. `Предмет`
2. `Тип`
3. `От класса`
4. `До класса`
5. `Действия`

## 5. Поведение строк

### Обычная строка
- `Предмет`: только чтение.
- `Тип`: badge, только чтение.
- `От класса` и `До класса`: только чтение.
- `Действия`: иконка удаления `Trash`.

### Редактируемая строка
В режим редактирования переводится только одна конкретная строка.

Можно редактировать:
- `minGrade`
- `maxGrade`

Нельзя редактировать:
- `subject`
- `type`

Если администратор понимает, что предмет у записи выбран неверно:
- он не меняет его в этой строке;
- он удаляет запись;
- затем создаёт новую правильную запись.

Это упрощает модель интерфейса и снижает риск случайного искажения уже используемых связей.

### Как запускается редактирование
Редактирование начинается **не по кнопке `Изменить`**, а по прямому взаимодействию с ячейкой:
- клик по ячейке `От класса` переводит в редактирование `minGrade`;
- клик по ячейке `До класса` переводит в редактирование `maxGrade`.

Это делает интерфейс быстрее и убирает лишнее действие там, где визуальная разница между просмотром и редактированием минимальна.

---

## 6. Создание новой компетенции

Создание должно быть встроенным и быстрым.

### Механика
1. Администратор нажимает `[+ Добавить предмет]`.
2. В верхней части таблицы появляется новая inline-строка.
3. Фокус сразу попадает в поле выбора предмета.

### Поля новой строки
1. `Предмет`
   - `Select` или combobox;
   - показывает все допустимые `Subject`, включая `REGIME`.
2. `От класса`
   - числовое поле.
3. `До класса`
   - числовое поле.

### Поведение
- `Tab` переводит фокус по полям.
- `Enter` сохраняет запись.
- `Esc` отменяет создание.

### Валидация
- предмет обязателен;
- `minGrade <= maxGrade`;
- нельзя создать вторую запись для того же преподавателя и того же предмета;
- если такая связь уже существует, показывается ошибка:
  - `Этот предмет уже назначен преподавателю`

### После успеха
- строка переходит в обычный read-state;
- список обновляется без перезагрузки страницы;
- сверху может появиться toast:
  - `Компетенция добавлена`

---

## 7. Редактирование существующей записи

Редактирование должно быть точечным и локальным.

### Запуск
- клик по ячейке `От класса` или `До класса`.

### Что редактируется
- только `От класса`
- только `До класса`

### Что не редактируется
- `Предмет`
- `Тип`

### Управление
- `Enter` сохраняет;
- `Esc` отменяет изменения;
- при сохранении строка возвращается в обычный вид.

### Валидация
- `minGrade <= maxGrade`;
- диапазон не должен быть пустым;
- если логика системы в будущем запретит определённые диапазоны, ошибка показывается прямо в строке.

---

## 8. Удаление

Удаление запускается по иконке `Trash` в строке.

### Базовый сценарий
- по нажатию открывается `AlertDialog`.

### Важный смысл
Удаление здесь является не только destructive-действием, но и официальным способом “исправить предмет”, если он был выбран неверно.

Поэтому текст подтверждения должен быть прямым:
- `Удалить связь преподавателя с предметом "Математика"?`

### Ограничения
Если в будущем появится логика, при которой связь уже используется в расписании или валидации генератора, можно добавить предупреждение, но в первой версии удаление допустимо.

---

## 9. Empty States

### А. У преподавателя нет компетенций
- Пустое состояние внутри таблицы.
- Текст:
  - `У преподавателя пока не назначено ни одного предмета`
- Подпись:
  - `Добавьте предметы и диапазоны классов, чтобы система могла учитывать этого преподавателя в учебном плане и расписании`
- Кнопка:
  - `[+ Добавить первый предмет]`

### Б. Фильтр или поиск ничего не нашли
- Иконка поиска.
- Текст:
  - `Ничего не найдено`
- Действие:
  - `Сбросить фильтры`

## 10. Использование в системе

Вопрос отображения `Где используется` пока **не зафиксирован окончательно**.

На текущем этапе спецификация закрепляет только безопасную рамку:
- экран в первую очередь показывает список компетенций преподавателя;
- он не должен превращаться в перегруженный обзорный дашборд;
- связи с другими модулями здесь точно не редактируются.

Что можно добавить позже отдельным решением:
- обзорный индикатор использования предмета в требованиях групп;
- краткую справку о наличии совместимых кабинетов;
- отдельный вторичный блок или popover с деталями.

Пока колонка `Где используется` **не считается обязательной частью первой версии экрана**.

---

## 11. Ограничения и доменные правила

Для первой версии спецификации фиксируются такие правила:
- один `TeacherSubject` на пару `teacherId + subjectId`;
- диапазон классов всегда непрерывный;
- `REGIME`-предметы допустимы;
- изменение предмета в существующей записи запрещено;
- исправление неверного предмета делается через удаление и создание новой записи.

Сервер должен поддерживать эти ограничения независимо от UI.

---

## 12. Server Actions и требования к данным

Минимально ожидаемые операции:
- `getTeacherSubjectsAction(teacherId, filters)`
- `createTeacherSubjectAction(data)`
- `updateTeacherSubjectAction(teacherId, subjectId, data)`
- `deleteTeacherSubjectAction(teacherId, subjectId)`

Валидация на сервере:
- учитель должен существовать;
- предмет должен существовать;
- `minGrade` и `maxGrade` должны быть валидными числами;
- `minGrade <= maxGrade`;
- нельзя создать дубликат пары `teacherId + subjectId`;
- `updateTeacherSubjectAction` не должен позволять менять `subjectId`.

---

## 13. Резюме для разработчика

- Это отдельная admin-страница, открываемая со страницы пользователей.
- Экран приоритетно обзорный, но допускает локальные правки.
- Не нужен ни `Sheet` профиля, ни глобальный edit mode, ни тяжёлые диалоги.
- Inline-редактирование применяется только к диапазону классов.
- Предмет в существующей записи не меняется: только удалить и создать заново.
