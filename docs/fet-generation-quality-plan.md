# План улучшения качества генерации FET

Этот документ фиксирует рабочий план по улучшению генерации недельного шаблона расписания через FET.

Правило источников: код и схема БД важнее старых документов. Документы используются как контекст и место фиксации решений.

## 1. Текущая картина в коде

Основной pipeline находится в `src/features/fet`:

- `collect-input.ts` читает `GroupSubjectRequirement`, `Group`, `Subject`, `TeacherSubject`, `TeacherAvailability`, `RoomSubject`, `Room`.
- `preflight.ts` проверяет кратность длительностей, наличие правил для `REGIME`, наличие учителя.
- `build-core-activities.ts` делает первый pass для `ACADEMIC`, `Завтрак`, `Обед`.
- `build-full-activities.ts` фиксирует найденный core и добавляет допы/остальные режимные занятия.
- `fet-xml.ts` строит `.fet` XML.
- `runner.ts` запускает `fet-cl`.
- `importer.ts` импортирует `*_activities.xml`.
- `generate-weekly-template.ts` транзакционно заменяет `WeeklyScheduleTemplate`.

Сейчас каждая `GroupSubjectRequirement` почти напрямую превращается в отдельные FET activities. После импорта все строки пишутся как `deliveryMode = DIRECT_GROUP`.

Статус на 2026-06-01:

- схема `regime pass -> full pass` заменена на `core pass -> full pass`;
- core pass включает `ACADEMIC`, `Завтрак`, `Обед`;
- full pass не создает `ACADEMIC` заново, а фиксирует core activities;
- допы получают hard lower bound после academic core-блока своей аудитории;
- `ElectiveGroupClassLink` собирается в FET input и используется для optional-групп во втором проходе;
- optional-группы получают `ConstraintActivitiesNotOverlapping` с завтраком/обедом открытых классов;
- предметы с `SUBJECT_SUBGROUP` не генерируются как занятие всего класса по тому же предмету;
- sibling-подгруппы одного класса и предмета получают разных преподавателей;
- sibling-подгруппы одного класса и предмета связываются soft `ConstraintActivitiesSameStartingTime`;
- импорт optional/shared-семантики еще не реализован: строки по-прежнему пишутся как `DIRECT_GROUP`.

## 2. Что уже есть в доменной модели, но почти не используется FET

В `prisma/schema.prisma` уже есть более богатая модель, чем использует FET-слой:

- `ScheduleDeliveryMode`: `DIRECT_GROUP`, `ELECTIVE_GROUP`, `SHARED_CLASSES`.
- `ElectiveGroupClassLink`: связь optional-группы с классами, для которых она открыта. Уже используется во full pass для lower bound и запрета пересечения optional с питанием открытых классов.
- `WeeklyTemplateOpenClass`: проекции optional-карточки в строки классов.
- `WeeklyTemplateCoveredClass`: покрытые классы для общего required/regime-занятия.
- `AttendanceLoadMode`: `DELIVERY_GROUP_SIZE`, `FULL_CLASS_SIZE`, `AFTERSCHOOL_COEFFICIENT`.
- `Subject.defaultAttendanceLoadMode`.
- `Room.seatsCount`.
- `TeacherAvailability` со статусами `AVAILABLE`, `PREFERRED`, `UNAVAILABLE`.

Текущий FET-экспорт этим почти не пользуется:

- комнаты экспортируются с `Capacity = 0`;
- student sets экспортируются с `Number_of_Students = 0`;
- `AVAILABLE` и `PREFERRED` игнорируются;
- отсутствие доступности учителя не считается недоступностью;
- optional-группы не пишутся обратно как `ELECTIVE_GROUP` с `openClasses`;
- общие required/regime-занятия не пишутся как `SHARED_CLASSES`;
- `breakDuration` превращается в FET `ConstraintMinGapsBetweenActivities` для пересекающихся групп детей.

## 3. Принятые доменные правила для FET

Эти правила исходят из текущего обсуждения и должны вести реализацию:

- Режимные моменты генератор может группировать сам с учетом вместимости помещений.
- `Завтрак` и `Обед` одного класса должны ставиться в одно и то же время в разные дни недели.
- На режимных моментах и обязательных допах можно объединять разные параллели, но возрастной градиент должен быть ограничен. Рабочее правило: разница классов не больше 2.
- Для питания допустимы временные волны, но не допускается превышение вместимости столовой.
- FET не создает доменные группы и не открывает запись. Он только размещает уже существующие группы и занятия.
- Доп по выбору может идти параллельно с обязательным допом.
- Доп по выбору может идти параллельно с режимным предметом.
- `breakDuration` - заданная перемена после урока. В FET применяется без послабления.
- Отсутствие записи доступности учителя означает недоступность.
- Если предмет ведется через `SUBJECT_SUBGROUP`, занятие этого предмета для всего parent-класса не должно попадать в FET.
- Подгруппы одного parent-класса по одному предмету желательно ставить одновременно и вести разными преподавателями.

## 4. Главные проблемы качества

### 4.1. Учительская доступность сейчас слишком мягкая

Сейчас в XML уходят только записи `UNAVAILABLE`. При реальных данных в dev-базе есть только `AVAILABLE`, поэтому FET фактически может ставить учителя куда угодно внутри дня.

Нужно перейти к модели:

- `AVAILABLE` задает разрешенные окна;
- `PREFERRED` задает мягкое предпочтение внутри разрешенных окон;
- `UNAVAILABLE` явно закрывает окна;
- отсутствие записи в сетке дня закрывает окно.

Первый hard-fix: построить `ConstraintTeacherNotAvailableTimes` как дополнение к объединению `AVAILABLE/PREFERRED`, с вычитанием explicit `UNAVAILABLE`.

### 4.2. Вместимость помещений сейчас не работает

`Room.seatsCount` и численность групп есть в БД, но XML сейчас пишет `Capacity = 0` и `Number_of_Students = 0`.

Нужно:

- читать `_count.studentGroups` для групп;
- экспортировать `Number_of_Students`;
- экспортировать `Room.Capacity = seatsCount`;
- считать расчетную аудиторию по `AttendanceLoadMode`;
- не давать FET выбирать помещение, если расчетная аудитория больше вместимости.

Для shared required/regime это особенно важно: расчет должен идти по сумме покрытых классов или по коэффициенту продленки.

### 4.3. Режимные моменты генерируются отдельно по классам

Сейчас `buildRegimeActivities` создает отдельную activity для каждого requirement класса. FET может поставить их рядом, но не понимает доменную идею "одно физическое занятие для нескольких классов".

Целевой результат:

- FET строит физические режимные слоты с несколькими покрытыми классами;
- импорт создает одну `WeeklyScheduleTemplate` с `deliveryMode = SHARED_CLASSES`;
- `coveredClasses` содержит классы, которые входят в общий слот;
- вместимость считается по полной численности для питания.

Нужно отдельно проверить на микрофикстуре FET-формат для activity с несколькими student sets. Если FET корректно принимает несколько `<Students>`, это предпочтительный путь. Если нет, нужен FET-only student set или другое моделирование.

### 4.4. Optional-группы теряют доменный смысл после импорта

В схеме есть `ElectiveGroupClassLink`, но генератор после импорта пишет только `DIRECT_GROUP`.

Нужно:

- для `ELECTIVE_OPTIONAL` использовать фактическую `ELECTIVE_GROUP`;
- брать открытые классы из `ElectiveGroupClassLink`;
- импортировать template как `deliveryMode = ELECTIVE_GROUP`;
- создавать `WeeklyTemplateOpenClass` для открытых классов.

FET не должен создавать optional-группы и не должен менять список открытых классов.

### 4.5. breakDuration передается в FET

UI/валидация уже знают `minimumBreakAfterMinutes`; FET теперь получает это правило как минимальный gap между activities пересекающейся student audience.

Реализованное правило:

- `breakDuration` передается в `FetActivity.breakAfterMinutes`;
- hard minimum для FET = ровно `breakDuration`;
- передачу `breakDuration` в FET можно временно отключить через `FET_ENABLE_BREAK_DURATION_CONSTRAINTS=false`;
- минимум переводится в periods через `FET_PERIOD_MINUTES`;
- constraint строится по группе детей, а не по кабинету;
- `CLASS` связывается со своими подгруппами;
- sibling-подгруппы не связываются;
- `ELECTIVE_GROUP` связывается с открытыми классами через `ElectiveGroupClassLink`;
- рядом с `Завтрак`, `Обед`, `Прогулка` перемена может быть 0 минут, если `FET_ALLOW_ZERO_BREAK_AROUND_REST` не равен `false`.

Ограничение FET симметрично для пары activities, поэтому оно выражает "между этими занятиями должно быть минимум N gaps". Для текущей модели это приемлемо как hard approximation: оно не зависит от кабинета и применяется только к пересекающейся группе детей.

### 4.7. Одинаковое время питания по дням

`Завтрак` и `Обед` одного класса связываются hard `ConstraintActivitiesSameStartingHour`.

Правило группировки:

- key = `groupId + subjectId`;
- завтраки класса связываются между собой;
- обеды класса связываются между собой;
- разные классы не связываются, чтобы сохранить волны столовой.

### 4.8. Предметные подгруппы вместо whole-class занятия

Если у класса есть `SUBJECT_SUBGROUP` по предмету, этот предмет моделируется только через подгруппы:

- `Group.subjectId` и требования подгрупп используются как источник признака "предмет разбит на подгруппы";
- `GroupSubjectRequirement` parent-класса по тому же предмету пропускается в core/full builders;
- preflight и выбор преподавателей также игнорируют такое parent-требование, чтобы не требовать лишнего учителя;
- sibling-подгруппы одного parent-класса и предмета получают разных преподавателей;
- FET получает soft `ConstraintActivitiesSameStartingTime` с весом `95`, чтобы одноименные занятия подгрупп вставали одновременно;
- sibling-подгруппы не получают `ConstraintMinGapsBetweenActivities` между собой, потому что это разные группы детей.

Ограничение: если у подгрупп разное количество занятий в неделю, same-starting-time строится только для попарно сопоставимых activities по порядку id. Лишние занятия остаются без этой мягкой связки.

### 4.6. Нет правил распределения уроков по неделе

Сейчас нет системного ограничения:

- не больше одного урока предмета в день;
- разносить повторяющийся предмет по разным дням;
- не перегружать один день;
- держать день компактным без больших окон;
- академические предметы до `FET_CORE_END_MINUTES`.

Нужно добавить мягкие constraints и пост-метрики:

- `ConstraintMinDaysBetweenActivities` для activity одной пары `audience + subject`;
- `ConstraintStudentsMaxGapsPerWeek` и, возможно, per-day вариант;
- дневные лимиты нагрузки класса;
- более жесткий утренний коридор для `ACADEMIC`, но с весами, которые не ломают сходимость.

## 5. План реализации

### Стратегия-кандидат: core-first generation

Идея: заменить текущую схему `regime pass -> full pass` на схему, где первый проход строит плотный каркас учебного дня, а второй проход добавляет допы.

Предлагаемый порядок:

1. Core pass:
   - `ACADEMIC`;
   - `REGIME` только для питания, минимум `Завтрак` и `Обед`;
   - hard/strong constraints на доступность учителей, кабинеты, вместимость, `breakDuration`;
   - compactness для классов включается именно здесь, потому что основная программа должна идти плотно.
2. Add-ons pass:
   - фиксирует все activities из core pass;
   - добавляет `ELECTIVE_REQUIRED`;
   - добавляет `ELECTIVE_OPTIONAL`;
   - добавляет остальные `REGIME`, если они не входят в core;
   - compactness для допов не навязывается или имеет сильно меньший вес.

Ожидаемый эффект:

- академические предметы перестают конкурировать с допами за поздние слоты;
- допы естественно вытесняются после основной программы;
- плотность основного блока можно настраивать отдельно от качества afternoon-блока;
- проще объяснить результат администратору: сначала школа, потом продленка/допы.

Риски:

- если в core pass включить завтрак/обед как обычную занятость всего класса, второй pass может запретить optional параллельно с режимом, хотя доменно это допустимо;
- если core pass слишком жестко уплотнить, он может заблокировать хорошие помещения/учителей для допов;
- если часть обязательных допов по смыслу должна быть "фоном" сразу после уроков, второй pass должен знать, что это не академический блок, но и не произвольная поздняя активность;
- fixed core activities во втором проходе повышают вероятность несходимости, если add-ons имеют слишком жесткие room/teacher/time constraints.

Промежуточное решение: первый проход должен фиксировать только те режимные события, которые действительно блокируют целый класс. Для режимных событий, параллельных optional, нужно либо моделировать частичную аудиторию, либо не делать их hard student-set blocker для optional-групп.

Реализованный первый срез:

- первый проход уже строит `ACADEMIC`, `Завтрак`, `Обед`;
- второй проход фиксирует core и добавляет допы/остальные `REGIME`;
- для direct/subgroup add-ons питание блокируется FET student-set constraints;
- для optional add-ons питание открытых классов блокируется явным `ConstraintActivitiesNotOverlapping`;
- optional add-ons стартуют после academic core-блока открытых классов по `ElectiveGroupClassLink`;
- compactness core-блока, вместимость и полноценная доступность учителей еще остаются следующими этапами.

Уточнения по домену:

- `Завтрак` и `Обед` блокируют весь класс. Параллельно с ними не может идти ничего.
- Для одного класса `Завтрак` и `Обед` крайне желательно ставить в одно и то же время каждый день.
- `Полдник` не входит в core pass.
- Обязательные допы должны идти строго после основной программы.
- Для core-блока важнее отсутствие окон, чем максимально раннее окончание.
- `breakDuration` применяется как жесткий минимум ровно `breakDuration`.
- Если следующий слот после предмета - `Завтрак`, `Обед` или `Прогулка`, минимальная перемена может быть `0`, потому что эти события считаются отдыхом.

### Этап A. Зафиксировать измеримость

Цель: перед изменениями получить стабильные метрики качества.

Сделать:

- добавить в артефакты генерации summary-файл с количеством activities, constraints, rooms, groups, warnings;
- после импорта прогонять существующую валидацию шаблона и сохранять результат в артефакты;
- считать метрики качества: нарушения вместимости, нарушения доступности, окна классов, количество gaps, отклонение режимных моментов по времени, run time FET.

Готовность: можно сравнить baseline и следующий прогон без ручного анализа XML.

### Этап B. Обогатить FET input

Сделать:

- расширить `FetGroup` количеством учеников и `subjectId`;
- расширить `FetRoom` вместимостью;
- расширить `FetSubject` `defaultAttendanceLoadMode`;
- собрать `ElectiveGroupClassLink`;
- собрать parent class для подгрупп;
- нормализовать доступность учителей в сетку FET.

Готовность: XML может содержать реальные размеры student sets и room capacity.

### Этап C. Исправить hard constraints

Сделать:

- генерировать недоступность учителей как complement к `AVAILABLE/PREFERRED`;
- не назначать учителя, у которого нет ни одного возможного слота для requirement;
- экспортировать вместимость rooms и students;
- отфильтровывать preferred rooms по вместимости;
- падать в preflight, если нет ни одного помещения, которое подходит по предмету и расчетной аудитории;
- после импорта валидировать результат теми же conflict/load правилами, что и UI.

Готовность: FET перестает выдавать расписания, нарушающие базовые ограничения школы.

### Этап D. Сохранить доменный смысл optional и shared

Сделать:

- `ELECTIVE_OPTIONAL` импортировать как `ELECTIVE_GROUP` + `WeeklyTemplateOpenClass`;
- required/regime, объединенные генератором, импортировать как `SHARED_CLASSES` + `WeeklyTemplateCoveredClass`;
- запретить объединение классов с разницей grade больше 2;
- запретить объединение `1` и `7` явно, даже если появится другая эвристика;
- не создавать доменные `Group` из FET.

Готовность: UI после генерации показывает не набор случайных прямых карточек, а реальные сценарии расписания.

### Этап E. Улучшить качество уроков

Сделать:

- добавить `MinDaysBetweenActivities` для повторяющихся lessons одного предмета;
- добавить ограничение "не больше одного урока предмета в день" там, где оно применимо;
- включить компактность дня с контролируемыми весами;
- добавить дневной максимум нагрузки класса;
- расставлять академические занятия до `FET_CORE_END_MINUTES`, но не ценой невозможности расписания;
- улучшить выбор учителя до FET: учитывать доступность, уже назначенную нагрузку и стабильность teacher для `group + subject`.

Готовность: расписание выглядит как школьное, а не просто как допустимое с точки зрения FET.

### Этап F. Итерационный подбор весов

Сделать:

- вести журнал экспериментов в `docs/fet-generation-runbook.md`;
- менять один фактор за прогон;
- хранить artifactId, env, summary и выводы;
- считать регрессии по метрикам.

Готовность: качество улучшается воспроизводимо, а не через разовые ручные догадки.

## 6. Вопросы, которые нужно подтвердить

1. `ELECTIVE_REQUIRED` по умолчанию должен считать аудиторию по `AFTERSCHOOL_COEFFICIENT`? В dev-данных сейчас все такие предметы имеют `DELIVERY_GROUP_SIZE`, но доменные документы говорят про коэффициент продленки.
2. `Прогулка` в dev-данных сейчас `ELECTIVE_REQUIRED`, а в доменных описаниях часто ведет себя как режимный предмет. Оставляем как обязательный доп или переводим в `REGIME`?
3. Для питания и режима объединение классов генератор делает сам. Нужно ли разрешать объединять классы разных смен/возрастов только по grade или еще по конкретному списку совместимых классов?
4. `breakDuration` применяется как hard minimum без допуска.
5. Academic + optional сейчас в UI-конфликтах запрещены. Подтверждаем, что optional может идти параллельно с required/regime, но не с academic?
6. Если у учителя есть только `PREFERRED`, но нет `AVAILABLE`, это доступное окно или мягкое предпочтение внутри уже доступного окна?

## 7. Ближайший рабочий срез

Самый полезный первый PR:

1. Расширить input студентами/вместимостью/открытыми классами optional-групп.
2. Исправить доступность учителей: отсутствие записи = недоступность.
3. Экспортировать реальные `Number_of_Students` и `Capacity`.
4. Импортировать optional как `ELECTIVE_GROUP` с `openClasses`.
