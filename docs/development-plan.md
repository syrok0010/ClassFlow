# ClassFlow — План разработки

## Обзор

ClassFlow — платформа для организации процессов дополнительного образования с модулем планирования расписания и встроенной CRM-системой. Проект построен на Next.js 16 + Prisma + PostgreSQL + Auth.js и включает интеграцию с FET-движком для автоматической генерации расписания.

Так как в команде **3 разработчика**, архитектура разделена на независимые модули (фичи). Это позволяет параллелить работу и четко разграничить зоны ответственности для защиты диплома (например: "Я делал модуль управления доступом и базовыми справочниками", "Я делал ядро расписания и drag-and-drop", "Я делал конвейер FET и статистику").

Ниже определены 5 промежуточных версий (milestone).

## Версия 0.1 — Фундамент и Базовый CRM (Foundation)
**Цель**: Поднять инфраструктуру проекта, настроить БД, роутинг, авторизацию и базовые CRUD-операции.

### Модуль 1: Инфраструктура и Авторизация (Разработчик А)
- **База данных**: Разворачивание PostgreSQL (Docker). Инициализация Prisma, перенос DBML-схемы в `schema.prisma`.
- **Авторизация**: Настройка Auth.js (NextAuth) с Credentials-провайдером.
- **Инвайт-система**: Механика "Админ создает пользователя → генерирует инвайт-ссылку → пользователь задает пароль".
- **Справочник пользователей**: Управление таблицей `User` (создание учителей, учеников, родителей) и привязка системных ролей (`Enum Role: ADMIN, USER`).

### Модуль 2: Пространство и Дисциплины (Разработчик Б)
- **Справочник помещений**: Управление `Building` и `Room` (создание, чтение, обновление, удаление через Server Actions). Интерфейс иерархии (Здание → Кабинет).
- **Справочник предметов**: Интерфейс для `Subject`. Разделение на типы: `ACADEMIC` (основные), `ELECTIVE` (допы), `REGIME` (прогулка, еда).
- **Матрица совместимости**: UI для связи кабинетов и предметов (`RoomSubject`), показывающий, где может проходить конкретное занятие (например, "Химия только там, где есть вытяжка").

### Модуль 3: Люди и Нагрузка (Разработчик В)
- **Справочник групп**: Управление таблицей `Group`. Иерархия (Класс → Подгруппа по английскому). Привязка детей (`StudentGroups`).
- **Компетенции учителей**: Интерфейс для `TeacherSubject` — какие предметы ведет учитель и для каких возрастных групп (`minGrade`, `maxGrade`).
- **Учебный план (Требования)**: Разработка сложного интерфейса (на базе `TanStack Table` + `TanStack Form`) для заполнения `GroupSubjectRequirement` (группа × предмет → кол-во уроков, длительность урока, необходимость длинной перемены).

**Результат v0.1**: Приложение запускается. Админ может залогиниться и полностью оцифровать школу (создать классы, предметы, кабинеты, раздать нагрузку учителям).

## Версия 0.2 — Кабинет учителя и Доступность
**Цель**: Создать UI для сбора ограничений (availability) от главных участников процесса — преподавателей.

### Модуль 4: Базовый портал учителя (Разработчик А)
- **Интерфейс учителя**: Личный кабинет с просмотром своей нагрузки и привязанных классов.
- **UI/UX сетки**: Визуальный компонент календаря-сетки (без сложного DnD расписания, только просмотр и выделение слотов).

### Модуль 5: Регулярное расписание доступности (Разработчик Б)
- **Матрица доступности**: Управление `TeacherAvailability` (чтение и запись блоков времени). Интерфейс позволяет учителю "закрашивать" блоки времени в неделе цветами: Зеленый (готов вести), Желтый (могу, но нежелательно), Серый (не могу).
- **Сохранение и валидация**: Server Actions для обработки этих матриц и сохранения сложных пересечений.

### Модуль 6: Исключения и Контроль (Разработчик В)
- **Разовые изменения**: Управление таблицей `TeacherAvailabilityOverride` (больничные, отпуска, отгулы на конкретные даты).
- **Дашборд администратора**: Сводный экран, визуализирующий "кто когда свободен" и подсвечивающий учителей, которые забыли заполнить свой график.

**Результат v0.2**: Школа собрала полные данные о том, когда учителя готовы работать, а когда нет.

## Версия 0.3 — Модуль генерации FET (Алгоритмическое ядро)
**Цель**: Автоматическая генерация "черновика" расписания (Шаблона).

### Модуль 7: Входной Пайплайн и Маппинг (Разработчик А)
- **Сериализатор XML**: Сбор всех данных из БД (Учителя, Кабинеты, Учебный план, Доступность) и их конвертация в специфичный XML-формат, который понимает движок FET.
- **Маппинг констрейнтов**: Перевод бизнес-правил (например, "Питание волнами по 15 минут", "Окна в расписании", "Учитель не может быть в двух зданиях") в терминологию FET-ограничений.

### Модуль 8: Интеграция CLI и Выходной Пайплайн (Разработчик Б)
- **Worker/Runner**: Запуск бинарного файла FET CLI на сервере (через `child_process`). Асинхронная отдача статуса генерации (progress-bar) на клиент.
- **Десериализация**: Парсинг сгенерированного FET XML-результата и сохранение его в базу данных в формат **`WeeklyScheduleTemplate`** (шаблон расписания).

### Модуль 9: Аналитика результата (Разработчик В)
- **Отчет о генерации**: Представление логов FET (какие soft-constraints алгоритму пришлось нарушить, почему не удалось поставить предмет).
- **UI "Черновика"**: Вывод сгенерированного `WeeklyScheduleTemplate` в виде read-only таблицы для первичной оценки администратором.

**Результат v0.3**: Админ нажимает кнопку "Сгенерировать". Сервер думает и выдает готовый шаблон расписания, учитывающий все 100500 математических ограничений.

## Версия 0.4 — Редактор Расписания (HITL Интерфейс Drag-and-Drop)
**Цель**: Интерактивный редактор для ручной шлифовки шаблона (Human-in-the-Loop) и управления конкретными записями. Данный этап — самая визуально сложная часть.

### Модуль 10: Интерактивный Drag-and-Drop Grid (Разработчик А)
- **Архитектура сетки**: Создание 2D сетки на базе `dnd-kit` и `Zustand`. Ячейки позиционируются абсолютно в зависимости от `startTime` и `duration`.
- **Перетаскивание**: Логика DnD карточек уроков. Обработка наложений, изменения длительности урока перетаскиванием (resize).

### Модуль 11: Real-time валидатор (Constraint Engine) (Разработчик Б)
- **Клиентский валидатор**: Система, которая в реальном времени при наведении карточки (`onDragOver`) проверяет:
    - Свободен ли кабинет? Подходит ли он для предмета?
    - Свободен ли учитель? Нет ли у него наложения?
    - Соблюден ли регламент столовой (для режимных моментов)?
- **Визуальная подсветка**: Окрашивание слотов в красный/зеленый на лету.

### Модуль 12: Система применения и оверрайдов (Разработчик В)
- **Применение шаблона**: Механизм конвертации `WeeklyScheduleTemplate` в фактические `ScheduleEntry` на заданный период дат (например, "Раскатать шаблон на сентябрь-октябрь").
- **Модуль экстренных замен**: UI для сценария "Учитель заболел". Система показывает затронутые `ScheduleEntry` на сегодня и предлагает свободных учителей нужной компетенции на замену.

**Результат v0.4**: Главная фича платформы готова. Расписание можно перетаскивать мышкой, система бьет по рукам за ошибки. Админ может управлять заменами.

## Версия 1.0 — Порталы и Полировка (Release)
**Цель**: Вывод конечных интерфейсов для потребителей расписания и подготовка к диплому.

### Модуль 13: Клиентские порталы (Распределение между командой)
- **Портал Ученика / Родителя**: Мобильно-адаптивный, максимально понятный интерфейс просмотра расписания на сегодня/неделю (Откуда забрать? Кто тьютор? В каком кабинете?).
- **Портал Учителя (Расписание)**: Отображение фактического расписания учителя с учетом всех замен.
- **Экспорт / Печать**: Генерация расписания в PDF / Excel для того, чтобы повесить на стену.

**Результат v1.0**: Платформа готова к внедрению и блестящей защите дипломного проекта. Каждый участник имеет весомые, технически сложные модули в портфолио.
