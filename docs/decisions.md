# Утверждённые архитектурные решения

> Этот документ фиксирует ключевые архитектурные концепции проекта ClassFlow и **причины**, по которым они были приняты. Точные структуры таблиц и связей см. в `docs/db-diagram.md`.

---

## 1. Технологический стек и Развертывание
- **Framework & UI**: Next.js 16 (App Router) + React 19. Выбрано для максимальной производительности SSR/RSC и использования Server Actions.
- **Стилизация**: Tailwind CSS 4. Используется нативный `@theme` для управления дизайн-системой.
- **База данных & ORM**: PostgreSQL + Prisma.
- **Аутентификация**: Auth.js (NextAuth). В начале — Credentials (placeholder-email + инвайт-коды от админа), позже — OAuth (Яндекс).
- **Деплой**: Docker для Production. Легкое развертывание в изолированной среде.

## 2. Бизнес-логика и Доменная модель

### Ролевая модель
Таблицы `Role` / `UserRoles` намеренно отсутствуют.
- **Системные роли**: Переданы в Enum `role` (ADMIN, USER) напрямую в таблице `User`. Обоснование: избегает лишних JOIN-ов при каждой проверке прав, так как системных ролей всего две и они фундаментальны.
- **Доменные роли (Parent / Teacher / Student)**: Определяются фактом существования записи `userId` в соответствующих доменных таблицах. Обоснование: Пользователь может комбинировать роли (например, быть и родителем, и учителем одновременно), авторизация строится на базе владения контекстом, а не глобальной роли.

### Режимные моменты и Тьюторы
Отдельные сущности для "Тьюторов" или "Режимных моментов" в БД не создавались.
- Воспитатели/Тьюторы моделируются как `Teacher`.
- Прогулки/обеды/сон моделируются как `Subject` с типом `REGIME`.
- **Обоснование**: Это позволяет использовать единый алгоритм генерации расписания и единую сетку без написания сложной кастомной логики для неакадемических событий (например, регламентированные походы в столовую "волнами" по 15 минут).

### Сетка звонков
В школе нет жестких фиксированных "уроков" (1-й урок всегда в 08:30 и т.д.).
- **Обоснование**: Малая школа с гибким расписанием для дошкольников и разных классов.
- **Решение**: У каждого события (`ScheduleEntry`) устанавливаются точные абсолютные `startTime` и `endTime`. Длина задается индивидуально.

### Шаблоны расписания vs Фактические занятия
Принята **двухслойная модель расписания**:
1. **`WeeklyScheduleTemplate`** — недельный цикл. Используется алгоритмом генерации (FET).
2. **`ScheduleEntry`** — физические записи на конкретную дату (инстансы).
- **Обоснование (из интервью)**: Расписание создается 2 раза в год как "идеал/черновик" (шаблон), но затем постоянно "едет" из-за замен и форс-мажоров. Администратор работает с шаблоном для генерации, а учителя/ученики видят только инстансы на конкретные даты. Это сохраняет историю и позволяет делать разовые изменения (override), не ломая базовую недельную сетку.

## 3. Интеграция с FET (Генерация расписания)
- **Концепция**: Подход "Система — рекомендатель".
- **Поток**: Сбор констрейнтов из БД → экспорт в FET XML → CLI генерация → импорт расписания в БД как черновик `WeeklyScheduleTemplate`.
- **Обоснование**: Заказчику нужен "черновик" с возможностью ручного шаманства. Интеграция делается не в первой фазе, сначала — удобный UI для сбора всех ограничений.

## 4. UI / UX и Клиентские библиотеки

На основе глубокого анализа рынка библиотек утверждены:

| Компонент / Библиотека | Обоснование выбора |
|---|---|
| **UI Kit**: `shadcn/ui` | Идеальная интеграция с Tailwind 4 CSS-переменными. Компоненты копируются локально, что безупречно подходит для архитектуры React Server Components. Не тянет тяжелые JS runtime (как Mantine). |
| **Формы**: `TanStack Form` + `Zod` | "Золотая середина". Отказ от `React Hook Form` в пользу TanStack, но с сохранением UI-примитивов `shadcn/ui`. Позволяет типо-безопасно работать со сложными табличными формами без падения производительности React. |
| **Таблицы**: `TanStack Table` | Headless решение для сложных CRUD-списков. Дает отличную интеграцию с TanStack Form (Inline-редактирование ячеек). |
| **Drag-and-Drop**: `dnd-kit` | В отличие от Pragmatic DnD, на мобильных устройствах (iPad учителей) перетаскивание срабатывает мгновенно по Pointer-событиям (без HTML5 long-press задержки). Идеально понимает 2D-сетки. |
| **Кэш и Данные (Data Fetching)**: `TanStack Query` | Критически важный мост между Server Actions и UI. Позволяет делать "Оптимистичные обновления" (Optimistic Updates) в drag-and-drop расписании, убирая задержку интерфейса при перетаскивании. |
| **State**: `Zustand` + `nuqs` (URL) | Zustand используется для клиентского UI-стейта, который нужно шарить между компонентами без Provider-обёртки (sidebar, DnD-матрица расписания и т.д.). Все фильтры/навигация — в URL (`nuqs`), серверные данные — в React Query. |
| **Виртуализация**: `TanStack Virtual` | Будет использоваться **только** для отображения огромных списков и таблиц (например, журналов и справочников). ⚠️ Категорически **запрещено** применять виртуализацию к сетке Drag-and-Drop расписания, чтобы не сломать расчет координат элементов. |
